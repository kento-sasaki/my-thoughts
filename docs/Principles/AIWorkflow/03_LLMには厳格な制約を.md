# LLM には厳格な制約を

[ドキュメントより実行可能な制約へ](./02_実行可能な制約へ.md)で、ルールは読ませる文章より実行できる制約にしたほうが効く、と書いた。この章はその続きだ。「実行可能な形にする」のが**手段**だとすれば、ここで書きたいのは、その手前にある**態度**の話になる。

人間相手なら窮屈すぎて続かないような厳格な制約を、LLM には敢えて課したほうがいい。制約は LLM にとってコストではなく、足場になる。私はそう思っている。

## 人間のコストと LLM のコストは非対称だ

良い設計の議論はたいてい「人間が書きやすく、読みやすいか」を基準にしてきた。厳格なレイヤー分け、例外を許さない命名、機械的なディレクトリ規則。こういうものは正しくても、人間には維持が重い。守るのに気力が要るし、急いでいるときほど崩れる。だから現実のコードベースは、どこかで規律と妥協のあいだに落ち着く。

LLM が書き手に回ると、この前提が崩れる。**実装量が苦にならない**のだ。人間なら「ここまで厳密にやるのは大変だ」と感じる規則も、LLM は例外なく、何度でも、淡々と適用する。チームみらいの開発で語られていた「人間が自分でやると大変でも、各レイヤーのルールを厳格に守らせるほうが LLM にとってはいい。実装量が苦にならないので、例外なくルールを適用して統一感を高めるほうに倒すほうが、トータルでスピードが出る」という観察は、まさにこの非対称性を突いている。

つまり、人間時代に「厳しすぎて割に合わない」と切り捨てていた制約のいくつかは、LLM 時代には**割に合う**ようになった。コストの構造が変わったのだから、設計の基準も変えていい。人間に優しい曖昧さより、LLM が踏み外しようのない厳格さを選ぶ。これは窮屈さの肯定ではなく、書き手が変わったことへの適応だ。

象徴的なのが、サーバー専用コードをクライアントから呼んでしまう類のミスだ。LLM に任せていると、実行時エラーになるこの取り違えがしばしば起きる。これを「気をつけて」と注意で防ごうとするのは人間向けの発想で、LLM には効きにくい。代わりに、パッケージ最上位で `server` と `client` を物理的に分け、`import 'server-only'` で誤用そのものを成立しなくする。**実行環境という分割軸をディレクトリと型で固定してしまえば、注意は要らなくなる。** 曖昧な指示より、踏み外せない構造のほうが、LLM には伝わる。

## 制約が実行可能なら、検証ループは LLM 自身で閉じる

なぜ厳格な制約がそこまで効くのか。鍵は、**厳格な制約はたいてい「実行可能」だ**という点にある。レイヤー違反は ESLint で弾ける。環境の取り違えは型と `server-only` で落ちる。規約違反はカスタムルールで検出できる。テストは通るか通らないかで答えを返す。どれも、LLM がその場で実行し、結果を読み取れる。

ここで効いてくるのが、モデル自身の自己検証だ。Claude Opus 4.7 は、自分の誤りをその場で捕まえ、書きながら直す挙動をネイティブに持つようになった。Anthropic は "It catches its own mistakes" "fixes its own code as it goes" と明言している。重要なのは、**テストを実行でき、その失敗を読み取れる環境があるとき、LLM は同一セッション内で自己修正を完結させる**という点だ。逆に言えば、検証可能な制約を用意していないと、この自己修正の能力は宙に浮く。直すべき失敗を、モデルが観測できないからだ。

だから厳格な制約は、LLM を縛るためのものではない。**LLM に「自分が間違えたかどうか」を教えるためのもの**だ。ESLint カスタムルールも、`server-only` も、型も、テストも、役割は同じ。即座に赤を返すフィードバック源を用意し、モデルの自己修正ループにそれを供給する。制約が実行可能であるほど、ループは速く、確実に閉じる。02 で書いた「実行可能な制約へ」が効く理由は、突き詰めればここにある。

## 制約をどこに置くかも、また設計だ

ただし、制約は無限に積めるわけではない。LLM のコンテキストは有限で、起動した時点で system prompt・メモリ・CLAUDE.md・各種定義が既に大半を埋めている。規約を読ませるたびにコンテキストは削られ、会話が進むほど最初の指示は薄まっていく。「全部の制約を常に効かせる」は、現実には不可能だ。

だから、制約を**どこに置くか**が、それ自体ひとつの設計判断になる。私の整理はこうだ。

- 常に効かせたい少数の原則は `CLAUDE.md` に常駐させる（その分コンテキストは常時消費される。だからこそ厳選する）。
- 特定の領域でだけ効けばいい規約は、path-scoped な `.claude/rules/` に逃がし、その領域に触れたときだけ読み込ませる。
- 手順としてまとまった専門知識は Skill にして、呼び出したときだけ展開する。
- 重い探索は subagent の別ウィンドウに分け、親のコンテキストを汚さない。

コンテキスト節約のコツは、個々の小細工ではなく、**知識ごとに「正しい置き場所」を選ぶ**ことに尽きる。そして最も強力な置き場所は、コンテキストの外、つまり ESLint や型や `server-only` のような**実行可能な制約**だ。それらはコンテキストを一切消費せず、`eslint` を走らせるだけで効く。制約を実行可能にすることは、検証ループを閉じるだけでなく、有限なコンテキストを節約する手でもある。

## 人間の重心は、書くことから決めることへ

厳格な制約を LLM に課し、検証を自動で閉じさせると、人間がやることは変わる。コードを一行ずつ書くことではなく、**何を制約とするかを決め、出てきたものをレビューすること**に移る。

制約の設計は人間にしかできない。どのレイヤー分けを正とするか、どの規約を実行可能にするか、何を `CLAUDE.md` に常駐させ何を捨てるか。これは「どう書くか」ではなく「何を正しさとするか」の判断で、まさに設計そのものだ。そして LLM が自己検証で潰せるのは「制約に照らして間違っているか」までで、「その制約自体が妥当か」は問えない。だから人間のレビューは消えない。むしろ、レイヤーの責務と実装が噛み合っているか、制約が目的に沿っているかを見る、**より上の層のレビュー**に純化する。

[表層から設計へ](./01_表層から設計へ.md)で、表層実装が安くなった分エンジニアの重心は構造とプロセスへ移る、と書いた。この章はその同じ移動を、別の角度から見ている。LLM に厳格な制約を課すというのは、書く仕事を手放し、**正しさを定義する仕事と、定義した正しさに照らして判断する仕事**を引き受けることだ。窮屈な制約を LLM に渡せるようになったぶん、私たちは「何を窮屈にすべきか」を考える側に回る。
