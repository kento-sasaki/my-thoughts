# ドキュメントより実行可能な制約へ


AI にコード規約や手順を守らせたいとき、最初に思いつくのは「ドキュメントに書いて読ませる」ことだ。`CLAUDE.md` に規約を書く、あるいは Markdown の規約集を参照させる。これは確かに効くが、限界がある。そしてその限界の越え方に、AI 時代のひとつの原則があると思っている。**ルールは「読ませる文章」より「実行できる制約」にしたほうが効く。**

## ドキュメントとして渡すことの限界

規約を Markdown でモデルに読ませる方式には、構造的な弱点が2つある。

1. **コンテキストを圧迫する** — 規約を読ませるたびに、限りあるコンテキストを消費する。
2. **会話が進むと薄まる** — モデルは直近の指示ほど優先する傾向があるため、会話を続けるほど、最初に読ませた規約は相対的に効かなくなっていく。

ドキュメントは「その場で読んで、やがて忘れられる」。人間の研修資料と同じで、読ませること自体は強制力にならない。

## 規約を「実行可能な制約」に落とす — ESLint カスタムルール

ここで発想を変える。規約のうち機械的に判定できるものは、文章ではなく **ESLint のカスタムルール**にしてしまう。たとえば「Button の中で Text コンポーネントを使わない」という規約は、`no-text-inside-button` のようなカスタムルールとして実装できる。

かつてカスタムルールの自作は面倒で割に合わなかったが、いまは AI がサクッと書く。実行可能な制約に落とすメリットは大きい。

- **コンテキストを圧迫しない** — 規約を読ませる必要がなく、`eslint` を実行するだけでいい。
- **AI が自己検証・自己修正できる** — コミット時に ESLint が走るので、AI は自分が生成したコードを自分で検証し、規約に従うよう直せる。

「読んで守る」から「実行して弾かれたら直す」へ。制約が実行可能になると、検証のループが自動で閉じる。これは第1部の[制約が設計を導く](../Engineering/03_制約が設計を導く.md)を、AI との協働に拡張した形だ。

（もちろん、すべての規約をルール化できるわけではない。自動化できる規約とできない規約を見極め、できるものから実行可能にしていく。）

## 手順を「呼び出せる専門知識」に落とす — Agent Skill

同じ考え方は、規約だけでなく**手順**にも当てはまる。常に適用したい方針は `CLAUDE.md`（常時読み込み・常時コンテキスト消費）に置く。一方、特定のタスクでしか使わない専門的な手順、たとえば PR レビューの観点・コミットの作法・特定フレームワークの生成手順は **Agent Skill** にする。

Skill はオンデマンドで読み込まれるため、必要なときだけコンテキストを消費する。たとえばコードレビューの観点を `/code-review` skill としてパッケージしておけば、一般的な観点に加えてプロジェクト固有の規約も踏まえたレビューを、呼び出したときだけ展開できる。

`CLAUDE.md` と Skill の使い分けは、「常に効かせたいか／必要なときだけ呼び出したいか」で決まる。知識を全部常駐させるのではなく、**呼び出せる形に分割して待機させる**。

## 共通するのは「知識を実行可能にする」こと

ESLint カスタムルールも Agent Skill も、やっていることは同じだ。**人間が読む前提のドキュメントを、機械が実行する前提の形に変換している。**

- 規約 → 実行可能な制約（ESLint）
- 手順 → 呼び出せる専門知識（Skill）

ドキュメントは知識を「保存」するが、実行可能な制約は知識を「強制」し、かつ AI に自己検証の手段を与える。だから、AI と協働する時代の知識は、読ませて期待するより、実行可能な形にして検証のループに組み込んでしまうほうがいい。これが、私が「ドキュメントより実行可能な制約へ」と言いたいことだ。
