# 表層から設計へ


AI がフロントエンド実装にもたらした最大の変化は、「表層を作るコストが劇的に下がった」ことだと思っている。マークアップやスタイリング、コンポーネントの見た目を組むこと。いわゆる表層実装は、もはや一番頑張るべき場所ではなくなった。

すると、仕事のボトルネックが移る。

## ボトルネックは「翻訳」に移った

これまでのフロントエンド開発は、Figma で作られたデザインをエンジニアが実装へ「翻訳」する工程を必ず挟んでいた。表層実装が重かった時代は、この翻訳コストは全体の中で目立たなかった。だが表層が安くなると、**Figma から実装への翻訳そのもの**がボトルネックとして残る。

ここで発想を変える。AI がコードの敷居を下げたのなら、デザイナーが Figma の段階で実装（View と Storybook）まで踏み出せばいい。そうすれば「Figma から実装へ翻訳する」工程自体が要らなくなる。職種の境界を、道具の力で越える。

## Figma は画面を完成させるが、意思決定を完了させない

なぜ「コードまで踏み出す」ことに意味があるのか。Figma は静止画だからだ。静止画は画面を完成させるが、意思決定までは完了させない。

- エラーバリデーションをどのタイミング（onChange / onBlur / onSubmit）で出すか
- Empty State はどう見えるか
- 文字数超過時の挙動
- 状態遷移の条件

こうした「インタラクションがないと気づかない仕様」は、Figma では曖昧なまま残せてしまう。

## コードは、書かないと動かない

ここで Storybook が効いてくる。Storybook で画面の全状態を Story として定義しようとすると、コンポーネントに渡す props や状態を**明示しなければ Story が作れない**。エラー状態・Empty State・ローディングと、すべての状態を列挙せざるを得ず、未定義の仕様が強制的に露出する。

問題を後工程ではなく前工程で発見する、いわゆる**フロントローディング**だ。「コードは書かないと動かない」という性質が、Figma で曖昧にできた穴を埋めさせる。

実際、コードを書けないデザイナーが Claude Code で Storybook を実装し PdM とウォークスルーした事例（CADDi, 2026-04）では、約 1.5 ヶ月で 31 画面・49 Story を作る過程で、**131 件の仕様穴・UX 改善を実装前に発見**したと報告されている。内訳は「Story に書くと未定義が露出する」仕様検討漏れ、「操作して初めて気づく」UX 違和感、「本番コードを参照する構造」による既存との不整合。象徴的なのはエラー設計の変化で、発想が「エラーを**表示する**」から「エラーを**防止する**」へ転換したという。

## エンジニアの重心は、どこへ移るのか

表層をデザイナーと AI が担えるようになると、エンジニアの価値は別の層に移る。

- 情報設計、UI/UX の設計、コンポーネント構造、データモデル設計といった、画面の裏にある**全体の構造**を作り込むこと
- そして、その構造を支える**開発プロセスの整備**（Storybook・Chromatic・CI・テスト・[CLAUDE.md や Agent Skill](./02_実行可能な制約へ.md) といった仕組み）

表層が安くなった分、私たちは「何を作るか」と「どう壊れないようにするか」により多くの時間を使える。これは引き算ではなく、重心の移動だ。表層から設計へ。AI 時代のフロントエンドは、この移動をどれだけ意識的に進められるかにかかっている、と思っている。
