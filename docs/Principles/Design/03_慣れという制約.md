# ユーザーの慣れという、見えない制約


設計には、仕様書に書かれた制約と、どこにも書かれていない制約がある。後者の代表が、ユーザーがこれまでの経験で身につけた**「慣れ」**だと私は思っている。[UI は認知フレームの設計](./01_UIは認知フレーム.md)では慣れを3つの視点の1つとして触れたが、ここではそれ自体を主役に据えたい。慣れは、設計を縛る見えない制約であり、そして同時に、作り手が使える（あるいは悪用できる）力でもあるからだ。

## 慣れは、明文化されない仕様である

ユーザーは、仕様書を読んでアプリを使い始めるわけではない。これまで触れてきた無数の UI から、暗黙のルールを身体に刻んで持ち込んでくる。

- 右のボタンは「進む」、左のボタンは「戻る」
- ハーフモーダルを下にスワイプすると閉じる
- テキストに下線があればリンク

どれも、どの仕様書にも書かれていない。それでもユーザーは当然のように期待する。だからこれらは、明文化されないまま設計を縛る制約として働く。私は、良い UI とは新しい体験を発明することではなく、**この見えない仕様を正確に読み取り、裏切らないこと**だと考えている。「直感的に使いやすい」という言葉の中身は、結局「ユーザーの慣れ通りに動く」ことに尽きる。

## 慣れを裏切ると、信頼が削れる

慣れの怖いところは、裏切ったときのコストがその場で見えないことだ。

私が実際に困った例がある。ある金融系サービスで使わなくなったアカウントを削除した。削除はあっさりできた。ところが数ヶ月後に再登録しようとすると、一度削除した電話番号とメールアドレスでは登録できない、と判明した。削除フローの中には確かに注意書きがあったらしいが、私はそれに気づけなかった。

なぜ気づけなかったのか。「削除は簡単に取り消せる」「消してもまた作り直せる」という**慣れ**を持っていたからだ。多くのサービスがそう作ってきたために、私はその前提で気軽に削除した。電話番号を再利用させない、という仕様自体は問題ない。問題は、ユーザーの慣れと食い違う重大な帰結を、慣れと同じ軽さで通過させてしまった設計のほうにある。慣れの裏切りは、その瞬間ではなく、ずっと後になって信頼として削れる。

## 注意は有限の資源で、慣れはそれを左右する

もう一つ、慣れには別の顔がある。**作り手が慣れを誘発し、注意を奪うこともできる**ということだ。

モーダルダイアログを考える。モーダルとは UI 部品の名前ではなく、ユーザーが特定の操作しかできないように制限された**状態**のことだ。その力は、ユーザーの注意を強制的に一点へ集める点にある。だから本当に重要な警告や、タスク継続に必須の入力にだけ使えば効く。

ところが、たいして重要でもない情報をモーダルで出し続けるとどうなるか。ユーザーは「またこれか」と**悪い意味で慣れ**、中身を読まず反射的に閉じるようになる。こうして、いざ本当に重要な警告を出したときには、もう誰も注意を払わない。さきほどの削除の注意書きを私が読み飛ばしたのも、根は同じだ。注意は有限の資源であり、それを浪費させる設計は、自分で自分の警告を無力化していく。

ここから私が引く原則はシンプルだ。**注意を強く奪う手段ほど、節約して使う。** モーダルはシンプルに、本当に重要なときだけ。複雑なタスクや、外の情報を参照しながら進めたいタスクは、ページやモードレスに逃がす。慣れを尊重するとは、ユーザーの注意という資源を大切に扱うことでもある。

## 制約は「何を実現したいか」から導かれる

では、慣れを尊重するとは常にモードレスを選ぶことか。そうではない。モードレスはユーザーの操作を分断せず創造性を解放する優れた思想だが、いつでも正解なわけではない。

たとえば DTx（治療用アプリ）では、治験プログラムに沿って正しい順序でタスクを進めてもらう必要がある。ユーザーの自由よりタスク完了の順序が重要なら、操作を制限するモーダル的な設計のほうが正しい。つまり、**ドメインの制約が UI パターンを決める**。慣れも、注意も、ドメインという土台の上で扱う変数だ。どちらのパターンが優れているかではなく、何を実現したいかが先にある。

## 慣れは「誰のための UI か」を静かに暴く

最後に、慣れにまつわる設計判断は、しばしば作り手の立ち位置を露わにする。

アカウント削除フローで、どちらをプライマリーボタンにするかを考えてみる。

| 配置 | 意味 |
| --- | --- |
| 「削除」をプライマリーに | ユーザーが望むアクションを優先 |
| 「戻る」をプライマリーに | サービスが望むアクションを優先 |

唯一の正解があるわけではない。だがこの一点の選択が、その UI が誰のために設計されているかを静かに語ってしまう。慣れを尊重しているように見せて、実はサービス側の都合へ誘導することもできる。だからこそ、作り手が「ユーザーと同じ慣れ」を自分のなかに持っているかが効いてくる。ユーザーが何を当然と感じるかを身体で分かっていなければ、尊重しているつもりで裏切ることになる。

慣れを読み、注意を節約し、それでも何を制限するかをドメインから決める。これは小手先の UI 改善ではなく、ユーザーの期待をどう扱うかという、設計の倫理そのものだと私は思っている。
