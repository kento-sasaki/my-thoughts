# UI は認知フレームの設計

<!-- 昇格元（private）: LT/drafts/2026-03-27_modal-modeless-and-user-habits/base.md（＋ 旧 Thoughts/Ideas/00_モーダル の主旨） -->

UI を「見た目」だと思っているうちは、デザインの半分しか見えていない。UI が本当に決めているのは、ユーザーが**何を期待し、何ができると感じ、どこに注意を向けるか**だ。つまり認知のフレームを決めている。ここでは、その見方を3つの視点から書く。

## 視点①：モーダルかモードレスか

「モーダル」は UI 部品の名前ではなく、**状態**のことだ。ユーザーが特定の操作しかできないように制限されている状態を指す。モーダルダイアログの力は、ユーザーの注意を強制的に一点に集める点にあり、危険は、使いすぎると認知負荷が増える点にある。

その対極が**モードレス**だ。モーダルがユーザーの操作を中断して特定の行動を強要するのに対し、モードレスはユーザーの操作を分断せず、創造性を解放する。紙にペンで書くときのような、対象を直接触っている感覚に近い。

ただし、モードレスが常に正解ではない。たとえば DTx（治療用アプリ）では、治験プログラムに沿って正しい順序でタスクを進めてもらう必要がある。ユーザーの自由よりタスク完了の順序が重要な場面では、モーダル的な思考のほうが正しい。**ドメインの制約が、UI 設計パターンを決める。** どちらが優れているかではなく、何を実現したいかが先にある。

## 視点②：ユーザーの「慣れ」という見えない制約

仕様書には書かれていないが、ユーザーが当然のように期待していることがある。

- 右のボタンは「進む」、左のボタンは「戻る」
- ハーフモーダルを下にスワイプすると閉じる
- テキストに下線があればリンク

これらは「慣れ」だ。そして慣れは、明文化されない設計上の制約として働く。慣れを裏切ると、ユーザーは混乱し、ときに実害を被る。たとえば「削除は簡単に取り消せる」という慣れに従って気軽に削除したら、数ヶ月後に同じ条件で再登録できなかった。こういう設計は、慣れの裏切りが信頼を損なう典型だ。

慣れは、設計判断が「誰のためのものか」も暴く。削除フローでどちらをプライマリーボタンにするかを考えてみる。

| 配置 | 意味 |
| --- | --- |
| 「削除」をプライマリーに | ユーザーが望むアクションを優先 |
| 「戻る」をプライマリーに | サービスが望むアクションを優先 |

正解が一つあるわけではない。だがこの一点の判断が、その UI が誰のために設計されているかを静かに語ってしまう。

## 視点③：直感が壊れる最大のポイント＝画面遷移

「直感的に使いやすい」とは、結局**ユーザーの期待通りに動く**ことだ。そして期待が壊れやすい場所には偏りがある。

タブ・ボタン・フォームのような部品は、UI ライブラリがある程度の挙動を保証してくれる。だから期待通りに動きやすい。一方で**画面遷移（ナビゲーション）は実装者が自分で設計する必要があり、ここが最も壊れやすい。**

ナビゲーションは単なる「画面の切り替え」ではなく、情報空間全体の設計だ。ドリルダウン（階層構造）、ステップ型（チュートリアル）、フラット型（タブバー）、ハブアンドスポーク（ホーム画面）、オーバーレイ（モーダルビュー）。どの構造を選ぶかで、ユーザーが「今どこにいて、どこへ行けるか」の認知が決まる。だからこそ、こうした中間行動は奇をてらわず教科書通りに作るのがいい。ユーザーの慣れを裏切らないことが、ナビゲーションでは特に効く。

## 共通するのは「ユーザーの期待をどう扱うか」

3つの視点は、結局ひとつのことを言っている。**UI とは、ユーザーの期待（＝認知のフレーム）を理解し、それを尊重するか裏切るかの選択である。**

- モーダル / モードレス → 操作を制限するか、自由にするか
- 慣れ → 見えない制約として設計に組み込むか、裏切るか
- ナビゲーション → 直感が壊れやすい場所で、期待をどう守るか

見た目を整えることは、このフレーム設計の結果にすぎない。そして「フレームを設計している」と自覚した瞬間、もうひとつの問いが立ち上がる。そのフレームは、ユーザーを良い方向へ形作っているのか。それを[道具が我々を形作る](./02_道具が我々を形作る.md)で考える。
