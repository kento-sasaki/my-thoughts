# 制約が設計を導く


制約は自由を奪うものとして嫌われがちだ。でも実際にコードを書いていると、良い制約はむしろ**設計を正しい方向へ導いてくれる**と感じる。守らなければならないルールにぶつかったとき、それを回避する小細工を探すか、ルールに素直に従える構造へコードを組み替えるか。後者を選ぶと、たいてい設計のほうが良くなっている。

## 制約が、責務の分割を教えてくれる

React Hooks には「条件分岐の中で呼んではいけない」という強い制約がある。条件によって必要な hooks が違うとき、この制約は厄介に見える。素直に書けないからだ。

```tsx
// 👎 制約違反。あるいは「全部呼んでおく」=不要なデータ取得が発生
const DailyRecord = ({ isShamPatient }) => {
  const { data: symptom } = useSymptomQuery()      // シャム患者には不要
  const { data: medication } = useMedicationQuery() // でもトップレベルでしか呼べない
  if (isShamPatient) return <ShamView />
  return <View symptomData={symptom} medicationData={medication} />
}
```

逃げ道として「条件に関係なく全部の hooks を呼ぶ」と、不要なデータ取得や副作用まで走る。ここで制約に**従う**方を選ぶと、自然と Container パターンに行き着く。条件を判断する層（Container）と、各条件で必要な hooks だけを呼ぶ層に分かれる。

```tsx
// 👍 Container が条件を裁く。各コンポーネントは自分に必要な hooks だけ呼ぶ
const DailyRecordContainer = ({ isShamPatient, ...props }) =>
  isShamPatient ? <ShamDailyRecord {...props} /> : <DailyRecord {...props} />
```

hooks のルールという制約が、「条件分岐と副作用は別々の責務として分けたら？」と問いかけてくれている。素直に従ってみると、責務が明確になり、テストもしやすくなる。（ただし、条件による違いが hooks ではなく表示内容だけなら、Container は不要だ。制約が要求していないところまで分割しない、というのも制約の読み方のうちだと思う。）

## 制約が、検知できる状態を作る

「不要な export をしない」も同じ構図だ。使われていない変数や関数を ESLint で検知する設定を入れていても、`export` してしまうと「どこかで使われているかもしれない」とみなされ、チェックをすり抜ける。

```tsx
// 👎 この画面でしか使わないのに export → 未使用検知が効かなくなる
export const FooHeader: FC = () => { /* ... */ }
```

公開範囲を必要最小限に絞る。つまり `export` という外向きの依存を増やさない制約を自分に課すと、ツールが「これは死んだコードだ」と検知できる状態が保たれる。制約を狭めるほど、自動チェックの効く範囲はむしろ広がる。これは[最小依存性](./01_最小依存性.md)の「不要な依存を持たない」とも地続きの話だ。

## 制約が、ロジックの輪郭を整える

`useEffect` を関連する処理ごとに1つにまとめる、という小さなルールもそうだ。同じフローに属する処理を別々の `useEffect` に散らすと、条件の優先順位が見えなくなる。

```tsx
// 👍 同じフローは1つの useEffect に。else if で優先順位が明示される
useEffect(() => {
  if (!programOpened) { showError('E18'); navigate(home) }
  else if (!previousSectionCompleted) { showError('E19'); navigate(home) }
}, [programOpened, previousSectionCompleted])
```

「関連するものは1箇所に」という制約を置くだけで、優先順位が `else if` の順序として現れ、不要なレンダリングも減り、ロジックの輪郭がはっきりする。

## 制約は「設計の問いかけ」として読む

ここまでの3つに共通するのは、**制約を「従うべき面倒」ではなく「設計の問いかけ」として読む**姿勢だ。

- hooks の順序ルール → 「条件と副作用は別の責務では？」
- export の最小化 → 「これは本当に外に出す必要が？」
- useEffect の集約 → 「これらは同じフローでは？」

つまり良い制約は、過去の良い設計判断を外に出して、毎回ゼロから考えなくて済むように再利用できる形にしてくれているのだと思う。制約に逆らって抜け道を探すと、たいてい設計は歪む。逆に、制約に素直に従える形を探すと、設計はむしろ整っていく。だから私は、制約を設計の敵ではなく、設計の案内役だと思って付き合っている。
