# 設計の原則

## これは何？

コンポーネントやモジュールをどう分け、どう繋ぐか。設計判断の根っこにある原則をまとめる。

個別の[コード規約](../../Practices/StyleGuide/README.md)が「何をすべきか（what）」なら、この章は「なぜそうするのか（why）」を書く。

## 通底する考え

設計とは、突き詰めれば **依存と責務をどこに配置するか** の判断だと思っている。動く・動かないを分けるのは機能の有無だが、良い設計・悪い設計を分けるのは、依存が意識的に配置されているかどうかだ。暗黙の依存、名前と実態のズレ、なし崩しに広がった責務。壊れやすいコードの裏には、たいていこのどれかがある。

この章では、その判断を3つの原則に分けて書く。

- [最小依存性：親に依存しないコンポーネント](./01_最小依存性.md) — 依存は最小に、そして明示的に。暗黙の依存こそが壊れやすさの源になる。
- [命名は言語](./02_命名は言語.md) — 名前は実態を語る言語であり、設計そのものを映す。
- [制約が設計を導く](./03_制約が設計を導く.md) — ルールや制約は窮屈さではなく、良い構造へ導く力になる。
- [恣意的な分割をやめ、責務で分ける](./04_責務で分ける.md) — 分割は「分けること」が目的化すると恣意的になる。責務に沿って分け、必要になってから分ける。
