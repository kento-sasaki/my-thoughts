# 恣意的な分割をやめ、責務で分ける


設計とは[依存と責務をどこに配置するか](./README.md)の判断だ、と私は思っている。ならば次に問うべきは「**どこで分けるか**」になる。コンポーネントをどの線で割るか。その線の引き方を、フロントエンドは長いあいだ「見た目 vs ロジック」という軸で考えてきた。だが私は、その分け方の多くは**恣意的**だったと考えている。

恣意的な分割と、責務に沿った分割は違う。前者は「分けること自体」が目的化していて、線が動く。後者は「ここで分かれている理由」を説明できる。Container/Presentational パターンの興亡は、この違いを React の歴史そのものが教えてくれた事例だと思っている。

## Container/Presentational は、動機は正しく、手段が恣意的だった

「賢いコンポーネント（Container）」と「馬鹿なコンポーネント（Presentational）」に分ける。2015 年に広まったこのパターンの**動機**——複雑な状態ロジックを見た目から引き剥がす——は、今でも正しい。問題は**手段**だった。

実際に書いてみると、「見た目 vs 動作」の境界は驚くほど曖昧だ。どこまでが見た目で、どこからがロジックか。線は人によって、日によって動く。そして Container はたいてい、データを受け取って下に流すだけの**単なる配線**になる。Mixins・HOC・Render Props も同じ課題（ステートフルなロジックの共有）への回答だったが、いずれも再利用のたびにツリーへ「意味のない層」を挿入する、いわゆる false hierarchy を生んだ。分けたことで、かえって構造が増えてしまう。

象徴的なのは、原著者の Dan Abramov 自身が 2019 年に撤回したことだ。「もうこの分割は勧めない。私が有用だと感じた主な理由は複雑なステートフルロジックの分離だったが、**Hooks は同じことを恣意的な分割なしに実現する**」。これは私にとって、ひとつの設計判断の答え合わせだった。分離という目的は生き残り、分離のための恣意的な器（Container でラップする習慣）が葬られた。

## Hooks は、層を増やさずに責務を分けた

Custom Hooks がしたことは、「ロジックの容れ物」をコンポーネントツリーから外したことだ。HOC や Render Props がラッパーで層を増やしたのに対し、Hooks は平らなまま、ロジックだけを抜き出せる。`useTodoSection()` のように、何の責務かを名前で語るフックに切り出せば、それは配線ではなく**意味のある分割**になる。

ここで大事なのは、撤回されたのは Container を切り出す**習慣**であって、Presentational の側——props を受けて UI を返す純粋なコンポーネント——はむしろ強化された、という点だ。捨てるものと残すものを取り違えないことが、この歴史から学ぶ最大の教訓だと思う。

## 「ちゃんと分けられたか」は、純粋な view が証明する

では、分割がうまくいったかを何で測るか。私は **view を Story にするのに hook を mock したくなったら負け**、という基準を使っている。

見た目を props だけの純粋関数に保てていれば、Loading / Error / Empty / HasData のすべての状態を props を差し替えるだけで Storybook に並べられる。その Story がそのまま Chromatic のビジュアルリグレッションのテストケースになる。逆に、view を描くのにフックのモックが要るなら、それは view にロジックが漏れているサインだ。漏れたロジックを hook 側へ押し戻す。`mock 不要で props だけで描けるか`が、分割の純度を測るリトマス試験紙になる。

つまり純粋な view は、美しさのためではなく、**責務がちゃんと分かれていることを機械的に検証可能にするため**に保つ。これは[制約が設計を導く](./03_制約が設計を導く.md)で書いた、「検知できる状態を作る」制約と同じ発想だ。

## ロジックは押し出さず、必要な場所で引く

view から逃がしたロジック——とりわけサーバー由来のデータ——をどう配るか。ここにも push と pull という分かれ道がある。

上流の Container が取得して Context で全体に**押し出す（push）**設計は、一見すっきりして見える。だが Section 単体テストに Provider が要り、Storybook で Context 全体のモックが要り、value 変化で関係ない部分まで再描画され……と、暗黙依存の代償を払い続けることになる。

TanStack Query のような仕組みがあるなら、各コンポーネントが必要な分を**引く（pull）**ほうが素直だと考えている。同じ `queryKey` を複数箇所で呼んでもリクエストは重複排除され、ネットワークは1回・キャッシュは1エントリで済む。だから各 Section が自前で `useQuery` を呼んでよい。これが「データ取得 Container」の正当な後継だ。

```tsx
export const useTodoSection = () => useQuery({ ...todoQueryOptions, select: d => d.items })
const TodoSection = () => { const { data } = useTodoSection(); return <TodoList items={data} /> }
```

Context をまったく使わないという意味ではない。Context は state manager ではなく**依存性注入の道具**だ、と割り切ると役割が定まる。上流で loading / error を一度だけ潰し、解決済みのデータだけを Context に入れる。子は可用性チェックなしに読める。これは「上流で取得済み」という暗黙依存を、Provider という辿れる形に**明示化**する DI だ。状態管理がしたいなら、Context ではなく selector で購読できる外部ストアを使う。

パフォーマンスも同じ筋で考える。再レンダリングの原因は4つ（自身の state・親の再描画・購読 Context の変化・使用 hook の変化）に還元でき、「props が変わると再描画する」は神話だ。だから `memo` を撒く前に、まず**状態の置き場所**を変える。state を使う部分を小さく切り出す、重い部分を `children` として渡す。置き場所で解けるものを、メモ化で塗り固めない。

## 分割の第一軸は、役割から実行環境へ移った

そして RSC が、この話の到達点をもう一段ずらした。コンポーネント分割の**第一軸**が、役割（Container/Presentational）でもレイヤー（Cell/View/Section）でもなく、**実行環境（Server / Client）**になる。`"use client"` がツリーを2つのモジュールグラフに割り、その境界がそのまま Container/Presenter の境界として機能する。

取得専用の層を切っていた動機は、ステートフルなロジックの分離だった。サーバーで `await` を直書きできるなら、その分離はもう要らない。データ取得の Container は server の `await` に吸収されて消える。Client は対話が要る葉だけの小さな島に保つ。「push から pull へ」の流れの延長線上で、server では `await` こそが pull になる。Hooks が恣意的な分割を消したのと同じ運動が、実行環境という新しい軸の上で続いている。

## コードベースの規模では、機能で分ける

ここまではコンポーネント単位の話だが、コードベース全体に視点を上げても結論は変わる。`components/` `hooks/` `containers/` という**技術種別**で切るのは、Container/Presentational と同じ恣意的な分割だ。種別フォルダは、ひとつの機能を理解するのに複数のフォルダを横断させる。

代わりに**機能（feature）単位**で切り、機能内にそのコードを自己完結させる。機能間は直接 import せず上位で合成し、依存は `shared → features → app` の一方向にだけ流す。健全さのテストはこうだ——ある feature を削除したら何が壊れるか。合成元の page だけが壊れるなら境界は効いている。他の feature に連鎖するなら、境界が破れている。

そして、この境界は規約に留めず `import/no-restricted-paths` で機械的に強制する。ここでも[制約が設計を導く](./03_制約が設計を導く.md)。ライブラリの選定は本質ではない。本質は**原則と境界**であって、ツールは差し替え可能だ。

## 分けないのではなく、恣意で分けない

誤解されたくないのは、これは「分割をやめよう」という話ではない、ということだ。やめるべきは恣意的な分割、つまり「分けること自体」が目的化した線引きだ。責務に沿った分割——名前で語れるフック、props だけで描ける view、削除テストに通る feature 境界——は、むしろ強化されている。

もうひとつ大切にしているのは、分離を**原則化しない**ことだ。「ロジックと見た目は常に分けるべき」と先に決めてしまうと、必要のない場所まで分割して、また恣意の罠に戻る。分離は必要になってから行う。早すぎる抽出も遅すぎる抽出も、どちらも避けたい。

線は、引けるから引くのではない。そこに責務の境目があるから引く。Container の興亡から RSC までを貫いて見えてくるのは、結局そういう一本の原則だと、私は思っている。
