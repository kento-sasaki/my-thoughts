# 命名は言語


命名は、設計のなかでいちばん安く、いちばん効く道具だと思っている。名前は単なるラベルではなく、そのコードが「何であるか」を周囲に伝える言語だ。名前と実態がズレた瞬間に、コードは嘘をつきはじめる。

## 名前は実態と一致していなければならない

`getCircle` という関数があるとする。中身は条件に応じて `<CheckBoldCircle />` などの JSX を返す。

```tsx
// 👎 関数の見た目だが、返すのは JSX.Element。実態はコンポーネント
const getCircle = (target: Date) => { /* ... return <Circle /> */ }
```

これは見た目こそ関数だが、`React.JSX.Element` を返す以上、実態は関数コンポーネントだ。`Circle` という名詞・パスカルケースで命名し `<Circle />` と呼べば、名前が実態（これはコンポーネントだ）を正しく語る。逆に言えば、命名規則は飾りではなく、**読み手が実態を推測するための文法**だ。だから私はフロントエンドの命名に一貫した規則を敷いている。

- コンポーネント・型はパスカルケースの名詞（`Button`、`SymptomRecordView`）
- 関数はキャメルケースの動詞（`handleSubmit`）
- Props として受け取るコールバックは `on + 動詞 + 目的語`、内部で定義するハンドラは `handle + 動詞 + 目的語`
- レイヤを名前に込める（`{Feature}Page` / `{Feature}Cell` / `{Feature}View`）

規則の細部より大事なのは、**名前を見ただけで種類と役割が分かる**状態を保つことだ。`data` や `info` のような曖昧な名前を避けるのも、ローマ字を避けるのも、すべて「名前に語らせる」ためだ。

## 抽象度が上がったら、名前を見直す

名前と実態のズレは、最初からではなく、育つ過程で生まれる。`RelaxationGoalCard` として作ったコンポーネントに、あとからタイトル・本文・色・アイコンを任意に差し替える props が足されていく。気づけば、それは「リラクセーションの目的を表すカード」ではなく、汎用的な `MessageCard` になっている。

```tsx
// 👎 名前は RelaxationGoalCard だが、実態は汎用 MessageCard
type Props = { title: string; body: string; icon: ReactNode; colorScheme: string }
```

私は、抽象度が上がったサインを「命名を見直せ」の合図として扱っている。

- `StackProps` などスタイル props を継承しはじめた
- `ReactNode` を props で受け取りはじめた
- props の数が増え続けている

このサインを見たら、実態に合った名前へリネームし、具体的なものは汎用コンポーネントを使って再定義する（`RelaxationGoalCard` は `MessageCard` を使った薄いコンポーネントに戻す）。

```tsx
// 👍 抽象度に応じて命名を分ける
export const MessageCard: FC<{ title: string; body: string; icon: ReactNode; colorScheme: string }> = (props) => { /* ... */ }
export const RelaxationGoalCard: FC = () => <MessageCard title="リラクセーションの目的" body="..." icon={<RelaxIcon />} colorScheme="blue" />
```

## 名前は、人と人の共通言語でもある

`RelaxationGoalCard` という名前を見て「これは汎用カードだ」と想像する人はいない。命名と実態の乖離は、コードの発見性を下げるだけでなく、チームの会話を壊す。デザイナーが「メッセージカード」と呼んでいるものを、コード上では別の名前で呼んでいると、設計の議論そのものがすれ違う。

だから命名は、コンパイラのためでも自分のためでもなく、**同じものを同じ名前で呼べる状態**を作るためにある。名前が揃えば、人とコードと設計が同じ言語で会話できる。これが、命名を「言語」と呼ぶ理由だ。
