# 命名は言語

命名は、設計のなかでいちばん安く、いちばん効く道具だと思っている。名前は単なるラベルではなく、そのコードが「何であるか」を周囲に伝える言語だ。名前と実態がズレるたびにコードの可読性は下がる。そしてそのズレの多くは、最初の命名ミスではなく、コードが育つ過程で静かに広がっていく。

## 名前は実態と一致していなければならない

例えば `getCircle` という関数が、条件に応じて `<Circle />` を返すなら、実態は関数ではなくコンポーネントである。ならば名詞・パスカルケースで `Circle` と命名し `<Circle />` と呼ぶ。命名規則 (コンポーネントは名詞、関数は動詞、コールバックは `on + 動詞`…) は飾りではなく、**名前を見ただけで種類と役割が分かる**状態を保つためにある。`data` や `info` のような曖昧な名前やローマ字を避けるのも、同じ理由による。

ただ、ここは間違えても気づきやすい。本当に厄介なのは、最初は正しかった名前が、育つうちに実態とズレていくケースだ。

## 抽象度が上がったら名前を見直す

最初は正しかった名前も、コードが育つと実態から少しずつズレていく。一段ずつは小さな変更なので気づきにくい。具体例で、抽象度が上がっていく過程をたどってみる。

「送料無料キャンペーン」を知らせるカードを作るとする。決まった色・タイポグラフィ・アイコン・文言で、常に同じものを表示する。最も具体的 ( = 抽象度が低く、使いまわしにくい) なので `FreeShippingCard` と名づける。

```tsx
// 文言も色もアイコンも固定。props を取らない＝最も具体的
const FreeShippingCard = () => <Box bg="blue.50"><InfoIcon />送料無料キャンペーン中</Box>
```

やがて「他の告知にも使いたい」と、タイトルと本文を任意の文字列に差し替えられるようにする。色とアイコンは固定のままだが、文言が任意になった時点で「送料無料」を表す名前ではなくなっている。実態は `InfoCard` だ。

```tsx
// 文言が任意に。色とアイコンはまだ固定
type Props = { title: string; body: string }
```

次に、お知らせだけでなく警告も出したくなり、情報の種類に応じて色とアイコンを切り替えられるようにする（情報なら青、警告ならオレンジ）。これはもう `MessageCard` と呼ぶべきものだ。

```tsx
// 種類で色とアイコンが決まる
type Props = { variant: "info" | "warning"; title: string; body: string }
```

さらにタイポグラフィもレイアウトも任意になり、どんな要素でも置けるようになると、残った共通点は「背景色・ボーダー・角丸・内側の余白がある」程度。ここまで来ると、もはや `Card` という最も抽象的な名前がふさわしい。

```tsx
// 中身もスタイルも任意。共通するのは枠だけ
type Props = { children: ReactNode } & StackProps
```

危ういのは、props だけがこうして育ち、名前が `FreeShippingCard` のまま取り残されるケースだ。一段ずつは小さな変更でも、積み重なれば名前と実態は大きく乖離する。 props に対して以下のような変化があるとき、それはコンポーネントの抽象度が上がっているサインかもしれない。

- `StackProps` などスタイル props を継承しはじめた
- `ReactNode` を props で受け取りはじめた
- props の数が増え続けている

このサインを見たら、実態に合った名前へリネームし、具体的なものは汎用コンポーネントを使って薄く再定義する。`FreeShippingCard` は、汎用化した `MessageCard` を使う薄いラッパーに戻せばいい。

```tsx
// 👍 抽象度に応じて命名を分ける
export const MessageCard = (props: { variant: "info" | "warning"; title: string; body: string }) => { /* ... */ }

export const FreeShippingCard () => <MessageCard variant="info" title="送料無料キャンペーン中" body="..." />
```

## 名前は、人と人の共通言語でもある

`FreeShippingCard` という名前を見て「これは汎用カードだ」と想像する人はいない。命名と実態の乖離は、コードの発見性を下げるだけでなく、チームの会話を壊す。デザイナーが「メッセージカード」と呼んでいるものを、コード上では別の名前で呼んでいると、設計の議論そのものがすれ違う。

だから命名は、コンパイラのためでも自分のためでもなく、**同じものを同じ名前で呼べる状態**を作るためにある。名前が揃えば、人とコードと設計が同じ言語で会話できる。
