# 最小依存性：親に依存しないコンポーネント


コンポーネントを設計するとき、私がいちばん気にしているのは「このコンポーネントは、自分の外側について何を**前提**にしているか」だ。

前提が多いコンポーネントは、一見すると便利に見える。だが前提とは、明文化されない依存のことだ。依存が暗黙のまま増えていくと、片方を変えたときにもう片方が静かに壊れる。だから設計の出発点は機能を足すことではなく、**依存を最小に、そして残った依存は明示的にする**ことだと考えている。

これは抽象的な標語ではなく、日々のコンポーネント実装で具体的に効いてくる。同じ原則がいくつもの場面で顔を出す。

## 親の実装を前提にしない

たとえば `<ul>` の中で使われることを前提に、`<li>` だけを返す `ListItem` を作るとする。これは「親が `List.Root` でラップしてくれる」という前提を子が持ってしまっている。

```tsx
// 👎 親が List.Root であることを知っている必要がある
export const ListItem: FC<{ label: string }> = ({ label }) => <List.Item>{label}</List.Item>
```

親からすれば子の内部実装を知って正しくラップしなければならず、子からすれば呼び出し元の構造を前提に書いている。どちらから見ても、親と子に余計な依存が生まれている。`List` 自身が `Root` から `Item` までを完結して持てば、その依存は消える。

## レイアウトの影響を受けない

同じことが見た目の世界でも起きる。コンポーネント全体を `<>...</>`（Fragment）で包んで返すと、中身が親のレイアウト指定の影響を受けてしまう。

```tsx
// 👎 FooForm を <Stack gap={4}> の中で呼ぶと、中の FormControl や Button が gap の影響を受ける
const FooForm = () => (
  <>
    <FormControl>...</FormControl>
    <Button>Submit</Button>
  </>
)
```

`FooForm` を `<Stack gap={4}>` の中に置くと、フォーム内部の間隔が親の `gap` に左右される。親は子の内部を知って `gap` を決めねばならず、子は親が適切に指定してくれる前提で書かれている。`<Stack>` のような器を自分で持てば、子は自分の見た目を自分で完結できる。

（Page やモーダルのように、そもそも上から `Stack` でラップされ得ないものは例外だ。原則は「暗黙の依存を生まないこと」であって、Fragment の禁止そのものではない。）

## 再生成の連鎖を断つ

依存は構造だけでなく、レンダリングのパフォーマンスにも現れる。`props` や `state` に依存しない関数をコンポーネントの内側で定義すると、再レンダリングのたびに関数が作り直され、`memo` 化した子まで巻き込んで再描画させてしまう。

```tsx
// 👍 props/state に依存しないなら、コンポーネントの外へ
const handleClick = () => console.log('clicked')
const MyComponent: FC<{ name: string }> = ({ name }) => <Child onClick={handleClick} />
```

「この関数は本当にこのコンポーネントの状態に依存しているか？」を問うと、依存していないものはコンポーネントの外へ追い出せる。依存していないものを内側に置かないことも、最小依存性の一部だ。

## バケツリレーを断つ — props drilling の解消

最小依存性をページ規模で考えると、`props` のバケツリレー問題に行き着く。Container/Presentational パターンで `Cell → View → Section` と組むと、`View` は各 `Section` に渡すデータを受け取って下に流すだけの「中継役」になりがちだ。`Section` が増えるたびに `View` の props は肥大化していく。`View` は本来の責務であるレイアウトを超えて、「どのデータをどの `Section` に配るか」という知識まで持たされてしまう。

これは `View` が各 `Section` のデータ構造に依存している状態だ。Section ごとに Context を分割し、`Cell` が Provider でラップして `Section` が自分の Context から必要なデータを取れば、`View` はレイアウトだけに集中できる。

```tsx
// 👍 View は中継しない。Section が自分のデータを自分で取る
const View = () => (
  <Stack>
    <SectionA />
    <SectionB />
  </Stack>
)
```

もちろんトレードオフはある。Section を単体で Storybook に載せるには対応する Provider が要る。だが重要なのは、「View が全 Section のデータを知っている」という暗黙の依存を、「各 Section が自分の関心だけを知る」という形に置き換えられることだ。依存の総量ではなく、依存の**置き場所**を設計している。

## 共通するのは「前提を持たない」こと

ここまでの4つは、見た目はバラバラだが同じことを言っている。

- 親の DOM 構造を前提にしない
- 親のレイアウトを前提にしない
- 不要なものに依存しない（＝コンポーネントの状態を前提にしない関数は外へ）
- 上流のデータ構造を前提にしない

コンポーネントは、自分の外側について知っていることが少ないほど、置き場所を選ばず、組み替えに強く、壊れにくい。「便利だから前提を持たせる」の逆を行く。これが、私が設計を考えるときの出発点になっている。
