# 最小依存性：親に依存しないコンポーネント

コンポーネントを設計するとき、私がいちばん気にしているのは「このコンポーネントは、自分の外側について何を**前提**にしているか」だ。

前提とは、明文化されない依存のことだ。依存が暗黙のまま増えていくと、片方を変えたときにもう片方が静かに壊れる。だから設計の出発点は**依存を最小に、そして残った依存は明示的にする**ことだと考えている。

これは抽象的な標語ではなく、日々のコンポーネント実装で具体的に効いてくる。

## 親の実装を前提にしない

たとえば `<ul>` の中で使われることを前提に、`<li>` だけを返す `ListItem` というコンポーネントを作るとする。これは「親が `<ul>` でラップしてくれる」という前提を子が持ってしまっている。

```tsx
// 👎 親が <ul> であることを知っている必要がある
const ListItem = ({ label }: { label: string }) => <li>{label}</li>
```

親からすれば子の内部実装を知って正しくラップしなければならず、子からすれば呼び出し元の構造を前提に書いている。どちらから見ても、親と子に暗黙的な依存が生まれている。

## 親の実装を前提にレイアウトしない

同じことが見た目の世界でも起きる。コンポーネント全体を `<>...</>`（Fragment）で包んで返すと、中身が親のレイアウト指定の影響を受けてしまう。

```tsx
// 👎 FooForm を <Stack gap={4}> の中で呼ぶと、中の FormControl や Button が gap の影響を受ける
const BarPage = () => (
  <Stack gap={4}>
    <FooForm />
  </Stack>
)

const FooForm = () => (
  <>
    <FormControl>...</FormControl>
    {/* ここの余白が親によって決まる */}
    <Button>Submit</Button>
  </>
)
```

`FooForm` を `<Stack gap={4}>` の中に置くと、フォーム内部の間隔が親の `gap` に左右される。親は子の内部を知って `gap` を決めねばならず、子は親が適切に指定してくれる前提で書かれている。

仮に  `FormControl` と `Button` の間の余白を親が指定できるようにしたいのであれば、 props で指定できる方が、依存が明示的なのでベターだと考える。

```tsx
const FooForm = ({ gap }: { gap: number }) => (
  <Stack gap={gap}>
    <FormControl>...</FormControl>
    <Button>Submit</Button>
  </Stack>
)
```

コンポーネントは、自分の外側について知っていることが少ないほど、置き場所を選ばず、組み替えに強く、壊れにくい。
