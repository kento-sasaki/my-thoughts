# 手順を Agent Skill にする

## 「全部 CLAUDE.md に書く」の問題

AI に何かを覚えさせようとするとき、最初に思いつくのは CLAUDE.md に書くことだ。プロジェクトのルール、コミットの作法、コードレビューの観点。これは正しい出発点だが、すべてを CLAUDE.md に積み上げていくと、やがて問題が見えてくる。

CLAUDE.md はセッション起動時に毎回読み込まれ、常時コンテキストを消費する。一般的な LLM のコンテキストは有限で、起動した時点で system prompt・メモリ・各種定義がすでに大半を占める。そこに長い CLAUDE.md が加わると、実際の作業に使えるコンテキストはさらに狭くなる。

もうひとつの問題は、CLAUDE.md が肥大化するにつれ、個々の指示が薄まることだ。モデルは直近の指示を優先する傾向があり、ファイルの序盤に書かれた指示ほど後半の会話では効きにくくなる。「全部 CLAUDE.md に書く」は、「全部常駐させる」ことを意味する。常駐させるには理由が必要だ。

発想の転換はシンプルだ。「常に効かせたい少数の原則だけを CLAUDE.md に置き、特定のタスクで使う手順は Skill に分ける」。これは[ドキュメントより実行可能な制約へ](../../Principles/AIWorkflow/02_実行可能な制約へ.md)で書いた「知識の置き場所を分ける」発想と同じだ。

## Skill とは何か — SKILL.md の構造

Claude Code の Skill は、`/skill-name` と呼び出したときにコンテキストへ展開される知識の単位だ。`.claude/skills/skill-name/SKILL.md` に定義し、呼び出されるまでコンテキストを消費しない。オンデマンド読み込みがポイントだ。

最小構成の SKILL.md はこうなる。

```markdown
# /code-review

## 目的

このプロジェクトの観点でコードレビューを行う。

## 前提

- 変更差分（`git diff`）を最初に確認する
- 深刻度は CRITICAL / HIGH / MEDIUM / LOW で分類する

## 手順

1. `git diff main...HEAD` で変更全体を把握する
2. セキュリティ観点を先に確認する（認証・入力値・外部 API 呼び出し）
3. コード品質を確認する（関数サイズ・ネスト深度・エラー処理）
4. 深刻度別にまとめて報告する

## 出力形式

### CRITICAL（マージ前に必ず対処）
- ...

### HIGH（できれば対処）
- ...

### MEDIUM 以下
- ...
```

タイトルに `/skill-name` を書くことで呼び出しと結びつく。「目的」「前提」「手順」「出力形式」の4要素が揃っていれば、AI は迷わず動ける。

## 実例 — `/code-review` Skill をパッケージする

コードレビューを例に考える。レビューの観点は2層ある。一般的な観点（セキュリティ・読みやすさ・テスト）と、このプロジェクト固有の観点（使ってはいけない API・特定の命名規則・レイヤー分けのルール）だ。

👎
```
# CLAUDE.md（悪い例）

コードレビューをするときは、以下の観点で確認してください。
1. セキュリティ: 認証・入力値...
2. 品質: 関数サイズ...
3. このプロジェクトでは Chakra UI の Text を Button の中で使わない...
（中略 — 50行）
```

毎回のセッションでこれが読まれ、コンテキストを消費する。コードを一行も書いていない時点で。

👍
```
# CLAUDE.md（良い例）

コードレビューの手順は /code-review を呼んで実行してください。
```

```markdown
# /code-review（SKILL.md）

## 目的

このプロジェクトのコードレビューを行う。

## 手順

1. `git diff main...HEAD` で変更全体を把握する
2. セキュリティ観点を確認する
3. プロジェクト固有の規約を確認する
   - Button の中で Text を使っていないか（`no-text-inside-button`）
   - ESLint が通っているか
4. 深刻度別に報告する
```

コードレビューの手順は Skill に閉じ、CLAUDE.md は呼び出しの一行だけになる。Skill は `/code-review` と呼んだときだけ展開される。

## CLAUDE.md と Skill の使い分け判断フレーム

どちらに置くかの判断基準は「常に効かせたいか、特定タスクのときだけか」だ。

| 置き場所 | 読み込みタイミング | 向く内容 |
|---|---|---|
| `CLAUDE.md` | 常時（セッション起動時） | プロジェクトの大方針・禁止事項・全体に効かせたい原則 |
| Skill | 呼び出し時（オンデマンド） | 特定タスクの手順・観点・出力形式 |
| `.claude/rules/`（path-scoped） | 該当ディレクトリ操作時 | 特定領域の規約 |
| subagent | 別セッション | 重い探索・長文読み込み・横断調査 |

「このルールは毎回必要か？」と問うて、答えが「特定の作業をするときだけ」なら Skill が向いている。置き場所の詳細な設計判断は[知識の置き場所を設計する](./03_知識の置き場所設計.md)にまとめた。

## Skill を育てる — 暗黙知をパッケージする

Skill の本当の価値は、「毎回同じ説明をしていることに気づく」きっかけになることだ。「新しいコンポーネントを追加するときはこの手順で」「PR を出す前にこれを確認して」——こういった説明をセッションのたびに書いていたら、それは Skill 候補だ。

一度 Skill に書いてしまえば、次回から `/skill-name` の一言で展開できる。観点や手順が変わっても、SKILL.md を更新するだけで以降の呼び出しに反映される。チームで使う場合も、ファイルをリポジトリに置くだけで共有できる。

暗黙知は言語化した時点で陳腐化が始まる、という見方もある。しかし Skill の利点は、「メンテが軽い」ことだ。変わった箇所だけ書き直せばいい。使われなくなった Skill はファイルを削除するだけだ。更新コストの低さが、Skill を育てる文化を維持しやすくする。

制約を「実行可能な形」にするという考えは[LLM には厳格な制約を](../../Principles/AIWorkflow/03_LLMには厳格な制約を.md)で詳しく書いた。Skill はその「呼び出せる専門知識」バージョンだ。

> **執筆時点の補足（2026-06 / Claude Code）**
> 本文で「Skill」と呼んでいるのは Claude Code の Agent Skill（`.claude/skills/` 配下に置く SKILL.md）を指している。呼び出し方（`/skill-name`）や読み込みタイミング・ファイルの配置規則はバージョンで変わりうる。現時点の仕様は Claude Code の公式ドキュメントを参照されたい。
>
> CLAUDE.md の読み込みタイミングや path-scoped rules の仕様も同様に、バージョンで変わる可能性がある。本文はあくまで設計の考え方（何を常駐させ、何をオンデマンドにするか）を中心に書いており、特定バージョンの実装詳細への依存は最小限にしている。
