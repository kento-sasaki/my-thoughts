# 知識の置き場所を設計する

## 全部の制約を常に効かせることはできない

AI に多くのことを覚えさせたいと思うのは自然だ。プロジェクトの方針、コーディング規約、レビューの観点、禁止パターン。これらをすべて効かせ続けられればいい。しかし現実には、LLM のコンテキストは有限だ。

セッションを起動した時点で、コンテキストは system prompt・メモリ・CLAUDE.md・各種設定に大半を使われている。そこに長い規約集や手順書を読ませると、実際の作業に使えるコンテキストはさらに狭くなる。「全部の制約を常に効かせる」は、構造的に無理がある。

コンテキスト節約の小細工——短くする、圧縮する、要約する——には限界がある。根本的な対処は「**知識ごとに正しい置き場所を選ぶ**」ことだ。この発想は[LLM には厳格な制約を](../../Principles/AIWorkflow/03_LLMには厳格な制約を.md)でも触れた。

## 4つの置き場所とその使い分け

知識の置き場所は大きく4つある。それぞれ読み込まれるタイミングとコンテキストへの影響が違う。

| 置き場所 | 読み込みタイミング | コンテキスト消費 | 向く内容 |
|---|---|---|---|
| `CLAUDE.md` | 常時（セッション起動時） | 常時消費 | 全体に効かせたい原則・禁止事項 |
| `.claude/rules/`（path-scoped） | 該当ディレクトリ操作時 | 操作時のみ | 特定領域の規約 |
| Skill | 呼び出し時（オンデマンド） | 呼び出し時のみ | 特定タスクの手順・観点 |
| subagent | 別セッション | 親に影響しない | 重い探索・横断調査 |

それぞれの判断基準は単純だ。「このルールは毎回必要か？」と問い、答えに応じて置き場所を選ぶ。

## CLAUDE.md — 常に効かせたい「原則の核」を置く

CLAUDE.md は常駐する。セッションが始まれば必ず読まれ、終わるまでコンテキストを占有する。だからこそ、ここに置くものは厳選する必要がある。

置くべきもの:
- プロジェクトの大方針（どういうアプリか、何を守る必要があるか）
- 全作業に必須のコンテキスト（ディレクトリの責務・技術スタック）
- 絶対に逸脱させたくない禁止事項

置くべきでないもの:
- 特定タスクでしか使わない手順（コードレビューの観点・コミット手順）
- 特定ディレクトリにしか関係ない規約（フロントエンドのコンポーネント規約をバックエンド作業中も読ませる必要はない）
- 長い参考情報（後で参照すればいいものをここに置くとコンテキストが膨らむ）

👎
```
# CLAUDE.md（悪い例）

## プロジェクト概要
...（5行）

## コーディング規約（React）
Button の中で Text を使わない。
...（20行）

## コーディング規約（TypeScript）
let より const を使う。
...（20行）

## コードレビュー手順
1. セキュリティを確認する
...（30行）
```

👍
```
# CLAUDE.md（良い例）

## プロジェクト概要
...（5行）

## 重要な制約
- コンポーネント規約: /frontend-rules を参照
- コードレビュー手順: /code-review を呼ぶ
```

CLAUDE.md は「索引」として機能させ、詳細は Skill や path-scoped rules に移す。

## `.claude/rules/`（path-scoped）— 領域別に読み込みを絞る

Claude Code では `.claude/rules/` 配下にファイルを置き、特定のファイルパスに操作が及んだときだけルールを追加で読み込む仕組みがある。フロントエンドの規約はフロントエンドのコードを触るときだけ、バックエンドの規約はバックエンドを触るときだけ読まれる、という分離ができる。

```
.claude/
  rules/
    frontend.md    # src/components/** にマッチするとき読み込む
    backend.md     # src/api/**、src/server/** にマッチするとき読み込む
    test.md        # **/*.test.ts にマッチするとき読み込む
```

この仕組みを使うと、「フロントエンドの Button / Text 規約」はバックエンドの作業中に読まれなくなる。必要なときに必要なルールが展開される。

path-scoped の設定は Claude Code の設定ファイルでパスパターンと結びつける。これにより、同じリポジトリに複数の技術スタックが共存していても、それぞれの規約が適切なタイミングだけ効く。

## Skill — 呼び出したときだけ展開する専門知識

Skill はオンデマンド読み込みの知識単位だ。`/skill-name` と呼ぶとコンテキストに展開され、呼ばなければ消費はゼロだ。コードレビューの観点、PR 作成の手順、特定フレームワークの生成手順——使うときだけ必要なものに向いている。

詳細は[手順を Agent Skill にする](./02_AgentSkill.md)にまとめた。

## subagent — 親のコンテキストを汚さない重い作業を分離する

大量のファイルを横断する調査、長いドキュメントの読み込み、外部リソースの網羅的な収集——こういった作業は親のセッションでやると、結果だけでなく途中の探索過程がすべてコンテキストに残る。それは後続の作業に使えるコンテキストを圧迫する。

subagent（別セッションの AI）に委ねると、探索の過程は別セッションに閉じ、親には「結果のサマリー」だけが返ってくる。親のコンテキストを温存しながら重い作業を完結できる。

典型的な用途:
- 複数ファイルにまたがるコードベースの把握
- 長いドキュメントからの情報抽出
- 広範囲の調査と選択肢の洗い出し

## 設計の原則 — 「最も強力な置き場所」は実行可能な制約

4つの置き場所をすべて使いこなしても、共通の限界がある。それは「コンテキストに置く」という性質だ。何かをコンテキストに置く限り、コンテキストを消費する。会話が長引くほど薄まる。

最も強力な置き場所は、コンテキストの外にある。ESLint ルール、型定義、`server-only` のような実行可能な制約だ。これらはコンテキストを一切消費しない。`eslint` を走らせるだけで効く。AI がそのエラーを読んで自己修正できる。

つまり、置き場所の設計は「4つのどれにするか」で終わらない。「実行可能な制約にできるか」が、その手前にある問いだ。機械的に検出できる規約は、読ませるのではなく ESLint や型で検出する。そちらに落とせるものは落とす。残った「文章としてしか伝えられない知識」を4つの置き場所に振り分ける。

ESLint カスタムルールの作り方は[規約を ESLint カスタムルールにする](./01_ESLintカスタムルール.md)に、実行可能な制約の思想は[LLM には厳格な制約を](../../Principles/AIWorkflow/03_LLMには厳格な制約を.md)に詳しく書いた。
