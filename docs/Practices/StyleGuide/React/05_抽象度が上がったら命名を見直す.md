# コンポーネントの抽象度が上がったら命名を見直す

## 規約

- 抽象度が上がったサインを検知したら、命名が実態に合っているか見直す。
- サインの例：`StackProps` などスタイル props を継承しはじめた／`ReactNode` を props で受け取りはじめた／props の数が増え続けている。

```tsx
// 👎 名前は RelaxationGoalCard だが、任意の title/body/icon/color を受け取る汎用カード
// 👍 実態に合わせてリネームし、具体は汎用を使って再定義
export const MessageCard: FC<{ title: string; body: string; icon: ReactNode; colorScheme: string }> = (p) => { /* ... */ }
export const RelaxationGoalCard: FC = () => <MessageCard title="リラクセーションの目的" body="..." icon={<RelaxIcon />} colorScheme="blue" />
```

命名と実態の乖離は発見性を下げ、デザイナーとエンジニアの会話をすれ違わせる。

## 例外

- 初期開発で再利用の見込みが不明なうちは、具体的な命名のままでよい。

→ なぜ：[命名は言語](../../../Principles/Engineering/02_命名は言語.md)
