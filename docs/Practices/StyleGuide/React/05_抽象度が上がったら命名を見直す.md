# コンポーネントの抽象度が上がったら命名を見直す

## 概要

- コンポーネントの抽象度が上がったサインを検知したら、命名がコンポーネントの実態に合っているか見直す。
- 抽象度上昇のサインとは以下のようなケース:
    - `StackProps` などのスタイル props を継承するようになった
    - `ReactNode` を props で受け取るようになった
    - props の数が増え続けている

## 👎 不適切な例

`FreeShippingCard` という名前のコンポーネントが、タイトルや本文を任意の文字列で変更でき、色やアイコンも切り替えられるようになっている。

```tsx
// NG: 名前は "FreeShippingCard" だが、実態は汎用的な MessageCard
type Props = {
  title: string          // 任意の文字列
  body: string           // 任意の文字列
  icon: ReactNode        // 任意のアイコン
  colorScheme: string    // 任意の色
  children?: ReactNode   // 任意の要素
}

export const FreeShippingCard: FC<Props> = ({ title, body, icon, colorScheme, children }) => {
  // ...
}
```

## 👍 適切な例

抽象度が上がった時点で、実態に合った命名にリネームし、具体は汎用を使って再定義する。

```tsx
// OK: 抽象度に応じた命名
// タイトルと本文だけ変更可能 → InfoCard
export const InfoCard: FC<{ title: string; body: string }> = ({ title, body }) => {
  // 色とアイコンは固定
}

// 色やアイコンも変更可能 → MessageCard
export const MessageCard: FC<{ title: string; body: string; icon: ReactNode; colorScheme: string }> = (props) => {
  // ...
}

// 元の具体的なコンポーネントは MessageCard を使って再定義
export const FreeShippingCard: FC = () => {
  return <MessageCard title="送料無料キャンペーン中" body="..." icon={<InfoIcon />} colorScheme="blue" />
}
```

## なぜこのルールが必要か

命名と実態の乖離は、コンポーネントの発見性と理解を阻害する。`FreeShippingCard` という名前を見て汎用的なカードだと想像する開発者はいない。
命名を見直すことで、コンポーネントの責務と抽象度レベルを明確に伝えられ、デザイナーとエンジニアが同じコンポーネント名で会話できるようになる。逆に命名と実態がすれ違ったままだと、デザイナーとエンジニアの会話もすれ違ってしまう。

→ なぜ：[命名は言語](../../../Principles/Engineering/02_命名は言語.md)

## 例外

- 初期開発段階で、まだ再利用の見込みが不明な場合は具体的な命名のまま進めてよい。
- 抽象度が上がるかどうかは、実際に再利用が必要になった時点で判断する。
