# フロントエンドの命名規則

> このノートは個々の規約スニペットではなく、命名の参照ドキュメント。`不適切な例 / 適切な例` 形式ではなくカテゴリ別にまとめる。

## 概要

- 命名はコードを読むための言語。レイヤ・役割・型が名前から読み取れるようにする。

## コンポーネント・型

- コンポーネント・型はパスカルケースの名詞（`Button`、`Checkbox`、`ColorPalette`）。
- props の型は `Props` または具体名（`ButtonProps`、`TherapyReflectionCardProps`）。
- レイヤを名前に込める：`{Feature}Page` / `{Feature}Cell` / `{Feature}View`。

```tsx
// Page: 機能名 + Page（ファイル: {Feature}Page.tsx）
const SymptomRecordPage = () => <SymptomRecordCell />
export default SymptomRecordPage

// Cell: 機能名 + Cell（QUERY / Loading / Failure / Success をエクスポート）
export const QUERY = gql`query FindSymptomRecordQuery { ... }`
export const Success = ({ data }: CellSuccessProps) => <SymptomRecordView />

// View: 機能名 + View（ファイル: {Feature}.view.tsx）
export const SymptomRecordView: FC<Props> = (props) => { /* UI 実装 */ }
```

```tsx
// 型もパスカルケースの名詞
export type ColorPalette = (typeof colorPaletteOptions)[number]
export type TherapyReflectionCardStatus = 'disabled' | 'inProgress' | 'completed'
```

## 関数・ハンドラ

- 関数はキャメルケースの動詞（`handleSubmit`）。
- props で受け取るコールバックは `on + 動詞 + 目的語`（`onCompleteLearning`、`onProceedToConfirmEmail`）。
- コンポーネント内で定義するハンドラは `handle + 動詞 + 目的語`（`handleSubmit`、`handleSelectAddActivityModal`）。
- Custom Hooks は `use + 機能名`（`usePatientSignUp`、`useActivityRecording`）。

## GraphQL

- Query：`Find + エンティティ + Query`（`FindSymptomRecordQuery`）。
- Mutation：`Create / Update / Delete + エンティティ + Mutation`（`CreateSymptomRecordMutation`、`UpdatePatientSettingMutation`）。

## ファイル拡張子

- `.tsx`：React コンポーネント / `.ts`：ロジック・型・ユーティリティ
- `.test.tsx`：テスト / `.stories.tsx`：Storybook / `.mock.ts`：モックデータ

## 避けること

- 略語の多用、日本語のローマ字表記、曖昧な名前（`data` / `info` / `item`）。
- 英語での明確な命名・機能を表す具体名・既存パターンとの一貫性を優先する。

## なぜこのルールが必要か

名前はコードを読む者にとっての言語であり、レイヤ・役割・型が名前から伝われば、実装を追わずに構造を把握できる。曖昧な名前や略語は読み手に推測を強い、一貫性のない命名は規約全体の信頼を損なう。
