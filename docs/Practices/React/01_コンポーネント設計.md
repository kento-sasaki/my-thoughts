# コンポーネント設計：恣意的な分割と目的のある分離


「Container/Presentational パターンはもう古い」とよく言われる。Dan Abramov 自身が 2019 年の追記で「もうこの分割は推奨しない」と書いた。では、データ取得や状態を抱える層と、見た目だけの層を分ける、という発想までまるごと捨てるべきなのだろうか。

私の結論はこうだ。**否定されたのは「恣意的な分割」であって、「目的のある分離」ではない。** コンポーネントを分けるとき、その境界が恣意的なのか目的があるのかを見分けられることが、いまのコンポーネント設計の肝だと思っている。

## 「恣意的な分割」とは何か

恣意的な分割とは、**UI 上の必然性がないのに、「ロジックを入れる容れ物」として技術的にコンポーネントを増やさざるを得なかった**ことを指す。

class コンポーネントの時代、state やライフサイクル（`componentDidMount` でのデータ取得など）は**コンポーネントの中にしか書けなかった**。だからステートフルなロジックを分離・再利用したければ、ロジックを置くためだけにもう一つコンポーネントを作るしかなかった。

```tsx
// 旧: ロジックを置くためだけに存在するコンポーネント
class UserProfileContainer extends React.Component {
  state = { user: null }
  componentDidMount() {
    fetchUser(this.props.userId).then((user) => this.setState({ user }))
  }
  render() {
    return <UserProfileView user={this.state.user} /> // DOM をほとんど出さない「容れ物」
  }
}
```

このコンポーネント境界は、画面構造上の意味ではなく**言語の制約**に従って引かれている。これが「恣意的（arbitrary）」の意味だ。HOC や render props は、さらに露骨に層を増やす（いわゆる wrapper hell）。

## hooks が無くしたもの

hooks は、**ロジックを「コンポーネント」ではなく「関数」に置けるようにした**。カスタムフックはレンダーツリーのノードではなく、ただの関数呼び出しなので層が増えない。Dan の "without an arbitrary division"（恣意的な分割なしに）が指すのは、まさにこの「ロジックのために別コンポーネントを作る強制」が消えたことだ。

証拠に、View を分離せず1コンポーネントに畳んでも、現代の React として全く問題なく成立する。

```tsx
'use client'
export function UserProfile({ userId }: { userId: string }) {
  const { user, isFollowing, onToggleFollow } = useUserProfile(userId) // ロジックは関数（hook）へ
  if (!user) return <Skeleton />
  return (
    <article>
      <h1>{user.name}</h1>
      <button onClick={onToggleFollow}>{isFollowing ? 'フォロー中' : 'フォローする'}</button>
    </article>
  )
}
```

## 旧パターンが束ねていた、2つの別々の分離

ここが本質だ。Container/Presentational パターンは、性質の違う2つの分離を**1つの「コンポーネント境界」として束ねて強制していた**。

- **(A) ステートフルなロジック ⇔ レンダリング の分離** — hooks が解決した部分。ロジックは hook（関数）へ移り、もうコンポーネントである必要がない。
- **(B) データに繋がったコンポーネント ⇔ props だけの純粋 View の分離** — Storybook や Visual Regression のための分離。昔も今も有効な、**目的のある選択**。

Dan が批判した「恣意的な分割」は (A) だ。(A) は言語の制約による強制であり、hooks がそれを不要にした。一方 (B) は、目的があって引く境界であり、強制されたわけではない。**両者を混同して「Container/Presentational はもう不要」と片付けると、(B) の価値まで捨ててしまう。**

実際、Storybook 公式は今も「presentational に保つ / container ロジックを分離する」原則を現役の推奨として記述している。純粋な View（props だけで見た目が決まり、副作用を持たない）を切り出すと、Story が `args` を渡すだけで完結し、Chromatic の Visual Regression が決定論的に回る。これは捨ててはいけない資産だ。

## 現代版：薄い Adapter は (A) ではなく (B)

現代版の「カスタムフック + 純粋 View」では、両者を繋ぐ薄い層（Adapter）が残ることがある。これは旧 Container と見た目こそ似ているが、別物だ。

| | 旧 Container | 新 Adapter |
| --- | --- | --- |
| ロジックの居場所 | 層の中に抱え込み、外に取り出せない | hook（独立した関数）として外にあり、単体でテスト・再利用できる |
| その層は必須か | 再利用するなら必須 | 1コンポーネントに畳めるので任意 |

つまり「恣意的な分割をしていた旧構造に戻った」のではなく、「見た目の階層は似ているが、技術的に強制された層という恣意性が消えた別物」だ。薄い層を引き続き "Container" と呼ぶこと自体は構わない。問題は名前ではなく、「ロジックを抱え込むことを強制された Container」という**存在**のほうにあった。

## だから、分割の前に「この境界は何のためか」を問う

設計判断としてのまとめはシンプルだ。コンポーネントを分けようとするとき、その境界が (A) なのか (B) なのかを自問する。

- ただロジックを置く場所が欲しいだけなら、それは恣意的な分割だ。コンポーネントではなく**関数（hook）**に置けばいい。
- 純粋な View を切り出して Story / Chromatic に載せたい、責務を見た目に閉じたい。目的があるなら、その境界は引く価値がある。

「コンポーネントを分ける／分けない」は手段であって目的ではない。目的のある境界だけを引き、恣意的な境界は引かない。これは第1部の[最小依存性](../../Principles/Engineering/01_最小依存性.md)、つまり「不要な層・不要な依存を持たない」とも地続きの話だ。

> **執筆時点の補足（2026-06 / React 19.2）**
> 「ロジックを関数や別実行環境に寄せ、View を純粋に保つ」という本文の原則は版に依存しないが、その具体的な実現手段は React のバージョンで変わりうる。執筆時点の事実を以下に切り分けておく。
>
> - **React Server Components (RSC)** は React 19 で stable（マイナー間で壊れない）。「Server Component でデータ取得 → 純粋 View に props / slot で渡す → 対話は Client Component」という形にすると、(B) の純粋 View 分離を実行環境の境界としても引ける。
> - ただし bundler / framework 実装向けの下位 API は semver に従わず、19.x のマイナー間で壊れうる（フレームワーク側はバージョン固定が推奨）。
> - よくある誤解の訂正：`"use server"` は Server Component の指示子では**ない**。Server Component 用の directive は存在せず、`"use server"` は Server Functions のためのもの。
>
